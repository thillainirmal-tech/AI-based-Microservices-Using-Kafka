SECURITY MODEL
─────────────────────────────────────────────────────────────────────────────
  - JWT is validated by the API Gateway (JwtAuthFilter) for all protected routes.
  - On success, the gateway injects two headers into every downstream request:
      X-User-Email : <email extracted from JWT subject>
      X-Trace-Id   : <UUID — generated per request for distributed tracing>
  - Downstream services derive user identity ONLY from X-User-Email.
    Never trust userId from the request body on protected endpoints.
  - Internal bank-service endpoints (/bank/debit, /bank/credit, /bank/refund,
    /bank/account/create) require header  X-Internal-Service: <caller-name>
    and are NOT exposed through the gateway. Never call them from Postman.
  - JWT expires in 24 hours. Re-login to get a fresh token.

PASSWORD POLICY (registration validation)
─────────────────────────────────────────────────────────────────────────────
  - Minimum 6 characters
  - Must contain at least one numeric digit (0–9)
  - Example valid password: "secret1"

FRAUD DETECTION THRESHOLDS (application.yml defaults)
─────────────────────────────────────────────────────────────────────────────
  high-amount-threshold    : 10000 INR   (amounts >= 10000 → FRAUD)
  max-transactions-per-day : 10          (velocity limit per user/day)
  impossible-travel-minutes: 5           (same user, 2 cities in < 5 min → FRAUD)
  review-confidence-min    : 0.4         (AI confidence 0.4–0.6 → REVIEW)
  review-confidence-max    : 0.6         (AI confidence > 0.6 → FRAUD)

──────────────────────────────────────────────────────────────────────────────
  4.1  GET TRANSACTION STATUS (combined fraud + payment verdict)
──────────────────────────────────────────────────────────────────────────────

  Method  : GET
  URL     : http://localhost:8081/api/transactions/{{transaction_id}}
            (direct to service — not available via local gateway)
  Headers :
    Authorization: Bearer {{jwt_token}}

  NOTE: Poll this endpoint every 2–5 seconds after submitting a payment.
  The fraud pipeline is async via Kafka. First response may be PENDING.

  Response Sequence:

  State 1 — Processing in progress (202 Accepted):
    {
      "transactionId"    : "f7a3c2d1-...",
      "fraudStatus"      : "PENDING",
      "overallStatus"    : "PENDING",
      "message"          : "Queued for fraud analysis",
      "retryAfterSeconds": 3,
      "timestamp"        : "2026-04-05T14:30:25.000"
    }

  State 2 — SAFE verdict, payment processing (202 Accepted):
    {
      "transactionId"    : "f7a3c2d1-...",
      "fraudStatus"      : "SAFE",
      "overallStatus"    : "PENDING",
      "message"          : "Fraud SAFE — processing payment",
      "retryAfterSeconds": 2
    }

  State 3 — COMPLETE (200 OK):
    {
      "transactionId"     : "f7a3c2d1-...",
      "fraudStatus"       : "SAFE",
      "fraudConfidence"   : 0.1,
      "fraudDetectionLayer": "RULE_ENGINE",
      "paymentStatus"     : "SUCCESS",
      "paymentMode"       : "BANK",
      "amount"            : 500.00,
      "payerEmail"        : "arjun.sharma@example.com",
      "payeeEmail"        : "priya.mehta@example.com",
      "payeeUpiId"        : "priyamehta@upi",
      "overallStatus"     : "COMPLETE",
      "message"           : "Final status",
      "timestamp"         : "2026-04-05T14:30:28.000"
    }

  State 4 — BLOCKED by FRAUD (200 OK):
    {
      "transactionId"      : "a1b2c3d4-...",
      "fraudStatus"        : "FRAUD",
      "fraudConfidence"    : 0.95,
      "fraudDetectionLayer": "RULE_ENGINE",
      "fraudReason"        : "Transaction amount 15000 INR exceeds high-risk threshold of 10000 INR",
      "overallStatus"      : "BLOCKED",
      "message"            : "Blocked (FRAUD)",
      "timestamp"          : "2026-04-05T14:30:26.000"
    }

  State 5 — REVIEW (needs human analyst):
    {
      "fraudStatus"        : "REVIEW",
      "fraudConfidence"    : 0.52,
      "fraudDetectionLayer": "AI",
      "overallStatus"      : "BLOCKED",
      "message"            : "Blocked (REVIEW)"
    }

  overallStatus Values Reference:
    PENDING      — Kafka consumer has not yet processed the event
    COMPLETE     — SAFE fraud verdict + payment SUCCESS
    BLOCKED      — FRAUD or REVIEW verdict; no money moved
    FAILED       — Bank debit/credit operation failed after SAFE verdict
    COMPENSATED  — Payer refunded after failed payee credit (saga rollback)
    CRITICAL     — Compensation itself failed; manual intervention required

================================================================================
  SECTION 7 — NOTIFICATION SERVICE
================================================================================

  The notification service has NO REST API. It is a pure Kafka consumer.
  It listens on the "notifications" Kafka topic and sends HTML emails.

  How it works:
    1. fraud-detection-service publishes a NotificationEvent to Kafka after:
       - A SAFE transaction completes (SUCCESS / FAILED / COMPENSATED)
       - A FRAUD verdict is issued (email alerts payer of blocked transaction)
    2. notification-service consumes the event and sends an HTML email via SMTP.

  NotificationEvent structure (published internally — for reference):
    {
      "transactionId": "f7a3c2d1-...",
      "userEmail"    : "arjun.sharma@example.com",
      "payeeEmail"   : "priya.mehta@example.com",
      "payeeUpiId"   : "priyamehta@upi",
      "amount"       : 500.00,
      "status"       : "SUCCESS",
      "message"      : "Your UPI payment of ₹500.00 was successful."
    }

  status values in NotificationEvent:
    SUCCESS   — payment completed; both payer and payee receive email
    FAILED    — bank debit/credit failed; payer is notified
    FRAUD     — transaction blocked by fraud engine; payer is warned
    COMPENSATED — refund processed; payer is notified

  How to verify notification delivery in Postman testing:
    1. Submit a payment via POST /api/upi/pay
    2. Poll GET /api/transactions/{id} until overallStatus = "COMPLETE"
    3. Check the email inbox configured in application.yml (SMTP_USERNAME)
       for an HTML email with subject "UPI Transaction Notification"
    4. Check notification-service console logs for:
         [NOTIFICATION] Email sent successfully to arjun.sharma@example.com

  SMTP configuration in notification-service application.yml:
    host : smtp.gmail.com   (or your configured SMTP host)
    port : 587
    username: ${SMTP_USERNAME}
    password: ${SMTP_PASSWORD}

  If you have not configured real SMTP credentials, the email will fail
  but the Kafka consumer will still log receipt of the event. Check logs
  for: [NOTIFICATION] Email sending failed — this is expected without SMTP.

  To verify Kafka message flow without email, use kafka-console-consumer:
    kafka-console-consumer.sh \
      --bootstrap-server localhost:9092 \
      --topic notifications \
      --from-beginning

================================================================================
  SECTION 10 — DEBUGGING GUIDE
================================================================================

  SYMPTOM: 401 Unauthorized on any protected endpoint
  ─────────────────────────────────────────────────────
  ✓ Check token is not expired (tokens last 24 hours)
  ✓ Ensure header is exactly: Authorization: Bearer <token>
  ✓ No extra spaces before/after token
  ✓ Re-run login to get a fresh token

  SYMPTOM: Transaction stuck at PENDING forever
  ─────────────────────────────────────────────────────
  ✓ Verify Kafka is running: kafka-topics.sh --bootstrap-server localhost:9092 --list
  ✓ Check transaction-service logs for: [TX-LIFECYCLE] SUBMITTED
  ✓ Check fraud-detection-service logs for: [FRAUD ENGINE] START
  ✓ Verify Redis is running: redis-cli ping → should return PONG
  ✓ Check fraud-detection-service logs for any deserialization errors

  SYMPTOM: 400 Bad Request on /api/upi/pay
  ─────────────────────────────────────────────────────
  ✓ payeeUpiId is required and cannot be blank
  ✓ amount must be > 0.01
  ✓ Verify payeeUpiId exists (use /bank/account/by-upi/{upiId} to check)

  SYMPTOM: /bank/balance returns 401
  ─────────────────────────────────────────────────────
  ✓ If calling directly (port 8084), the gateway has NOT injected X-User-Email
  ✓ Manually add header: X-User-Email: arjun.sharma@example.com
  ✓ This header is normally injected by the gateway from the JWT — add it yourself
    when bypassing the gateway for direct service calls

  SYMPTOM: Payment goes to FAILED or COMPENSATED
  ─────────────────────────────────────────────────────
  ✓ Check bank-service logs for debit/credit failures
  ✓ Verify payer has sufficient balance
  ✓ Verify payee UPI ID is registered
  ✓ Check MySQL connection: bank_db table accounts

  SYMPTOM: 403 on /bank/debit or /bank/credit
  ─────────────────────────────────────────────────────
  ✓ These are internal endpoints — they require X-Internal-Service header
  ✓ Add: X-Internal-Service: fraud-detection-service
  ✓ These are designed for service-to-service calls only

  SYMPTOM: Notification not received
  ─────────────────────────────────────────────────────
  ✓ Check notification-service logs for: [NOTIFICATION] Received
  ✓ If "Email sending failed" — SMTP credentials not configured
  ✓ Use kafka-console-consumer to verify the Kafka message was published:
      kafka-console-consumer.sh --bootstrap-server localhost:9092 \
        --topic notifications --from-beginning --max-messages 5

  SYMPTOM: Gateway returns 404 for /api/upi/** or /api/transactions/**
  ─────────────────────────────────────────────────────
  ✓ These routes are NOT in application-local.yml (only in docker profile)
  ✓ Solution A: Call transaction-service directly on port 8081
  ✓ Solution B: Add the missing routes to application-local.yml (see top of guide)
  ✓ Ensure gateway started with: --spring.profiles.active=local

  SECTION 12 — KAFKA TOPIC REFERENCE
================================================================================

  Topic               Service           Direction     Purpose
  ──────────────────────────────────────────────────────────────────────────
  transactions        transaction-svc   PRODUCE       UPI/legacy transactions
                      fraud-detection   CONSUME       Fraud analysis input

  transactions.DLT    fraud-detection   PRODUCE       Dead-letter queue (after 3 retries)

  notifications       fraud-detection   PRODUCE       Payment outcome events
                      notification-svc  CONSUME       Email delivery


  Consumer Groups:
    fraud-detection-group-v2  → fraud-detection-service (transactions topic)
    notification-group        → notification-service (notifications topic)


  Useful Kafka CLI commands (run from Kafka bin directory):

  List all topics:
    kafka-topics.sh --bootstrap-server localhost:9092 --list

  Inspect consumer group lag (is fraud-detection keeping up?):
    kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
      --group fraud-detection-group-v2 --describe

  Consume latest transactions messages (live monitoring):
    kafka-console-consumer.sh --bootstrap-server localhost:9092 \
      --topic transactions --from-beginning

  Consume DLQ (failed messages):
    kafka-console-consumer.sh --bootstrap-server localhost:9092 \
      --topic transactions.DLT --from-beginning
